# 運用ガイド：PIIマスキング

**ステータス**: Draft v0.1

---

## 1. 平常時

### 1.1 監視

Grafana ダッシュボード「PII Operations」：

- 検出件数（エンティティ別 × テナント別、過去24h）
- ブロック件数（テナント別、過去24h）
- スキャン時間 p50/p95/p99
- スキャンエラー率
- 検出率の前週比

### 1.2 アラート

| 名前 | 閾値 | 緊急度 | 通知先 |
|---|---|---|---|
| `pii_scan_p99_latency_high` | p99 > 100ms × 5分 | warn | Slack |
| `pii_scan_errors_burst` | errors > 5/分 | critical | Slack + Pagerduty |
| `pii_block_spike` | テナント別ブロック数が直近7日中央値の3σ超 | warn | Slack + 顧客担当 |
| `pii_no_detection_24h` | テナントの検出件数が24h ゼロ（普段ある場合） | info | Slack |

## 2. ポリシー変更運用

### 2.1 グローバルデフォルト変更

1. Gitリポにポリシー YAML をPR
2. レビュー2名以上（エンジニアリングリード + 法務担当）
3. ステージング環境で 24h 動作確認
4. 段階リリース：影響テナント10%→50%→100%
5. 変更後 7 日間は週次でメトリクス監視

### 2.2 テナント別オーバーライド

1. 顧客から書面（管理画面の申請フォーム）で依頼
2. 営業 or CS が法務承認を取得
3. 管理画面から適用
4. 操作ログ自動記録（操作者・時刻・前後値・理由）
5. 顧客にメールで適用完了通知

## 3. 検出精度の継続改善

### 3.1 定期レビュー（四半期）

- ゴールデンセット（200件）に対する Precision / Recall を計測
- 「マスクされたが本来PIIでない」候補のサンプリングレビュー
- 「マスクされなかったが本来PIIだった」事例の収集（顧客報告含む）

### 3.2 パターン更新

新パターン追加：
1. 失敗事例を test fixture に追加
2. recognizer に pattern 追加
3. ユニットテストで再現確認
4. 既存テスト全件パス確認
5. ステージング→本番リリース

### 3.3 false positive 報告窓口

顧客管理画面に「誤検知レポート」ボタン。クリックすると：
- 該当テキスト（マスク済み）
- 検出エンティティ
- 顧客コメント
が運用窓口に送付。**マスク前テキストは送付しない**。

## 4. 障害対応

### 4.1 PIIスキャナ障害

**症状**：`pii_scan_errors_total` が急増、リクエストが451で拒絶される。

**初動**：
1. Grafana で原因確認（spaCy ロード失敗、メモリ不足、特定パターンの ReDoS など）
2. 緊急時は `fail_open=true` を全テナントで一時有効化（**法務承認必須**）
3. 並行してホットフィックス開発

**事後**：
- インシデントレポート
- 顧客向け通知（影響範囲・期間・対応）

### 4.2 ReDoS（正規表現DoS）

- 起動時に再帰的バックトラックの危険なパターンを `re2` で検証（CI に組込）
- `regex` モジュールの `TIMEOUT` パラメータで個別マッチを 100ms 制限
- 異常に長い入力（>1MB）は事前拒絶

### 4.3 spaCy モデル更新

- 年次の新リリース確認
- マイナーバージョン更新は staging で 1 週間運用後、prod
- メジャー更新は性能テスト必須

## 5. 監査対応

### 5.1 個情委からの開示要求

1. 顧客の同意取得（プライバシーポリシーに従い）
2. テナント別 audit ログから当該期間の操作・検出履歴を抽出
3. **マスク済みデータ**を提出（生データは別途裁判所命令等が必要）

### 5.2 SOC2 / ISMS 監査

監査人に提示する資料：
- 本ドキュメント一式
- 月次運用レポート（過去12ヶ月分）
- 障害対応履歴
- 改善活動記録
- アクセス権限管理記録
- 変更承認ログ

---

## 6. オンボーディング SOP（新規顧客）

1. プラン選定 → 業界別パッケージ提案
2. デフォルトポリシー適用
3. 顧客側のサンプルデータで動作確認（PoC環境）
4. 必要に応じてオーバーライド申請
5. 本番切替
6. 初週は連日モニタリング
7. 月次レポートで定常運用へ移行
