# LLMプロバイダ規約コンプライアンス分析

**ステータス**: Draft v0.1 / **要弁護士＋プロバイダ事業窓口確認**
**作成日**: 2026-05-14

---

## 0. 重要な前提

Aegisは「**顧客のAPIキーを使って、顧客の代わりにLLMプロバイダAPIを呼ぶプロキシ**」である。これは各プロバイダの利用規約上、**慎重な確認が必要な構造**である。

- **顧客主体モデル（推奨）**：顧客がプロバイダと直接契約し、APIキーを発行。Aegisは「顧客の指示に従ってAPIを呼ぶ業務委託先」。
- **Aegis主体モデル（非推奨）**：Aegisが自社のAPIキーで顧客にLLMを再販。多くのプロバイダで規約違反になり得る。

本ドキュメントは**顧客主体モデル**を前提として、各プロバイダの規約を整理する。**規約は頻繁に変更される**ため、ローンチ直前と四半期ごとに再確認すること。

---

## 1. 一般原則：顧客主体モデルの根拠

「顧客のAPIキーを使う」構造は以下の理屈で整理：

1. **当事者は顧客とプロバイダ**：API契約は顧客とプロバイダ間。Aegisは契約当事者ではない。
2. **Aegisは業務委託先**：顧客から「自社AIアプリの観測・運用」を委託されている。
3. **データ処理は顧客の指示の範囲内**：顧客のリクエストを忠実にプロキシし、ログを取る。

これにより、再販規約・利用権限の譲渡禁止条項を回避できる可能性が高い。ただし**プロバイダの追加同意・通知が必要な場合がある**。

## 2. プロバイダ別分析

### 2.1 Anthropic（Claude API）

#### 規約根拠
- **Commercial Terms of Service**
- **Usage Policies**
- **Trust Center / DPA**

#### 要点
| 観点 | 内容 |
|---|---|
| サブプロセッサーとしての位置 | 顧客がAnthropicとDPAを締結している場合、顧客がAegisを「自社のサブプロセッサー」として位置付ける必要あり |
| データ学習利用 | API経由のデータはデフォルトで学習に使われない（規約・FAQで確認） |
| データリージョン | デフォルトは米国。AWS Bedrock経由なら東京リージョンも選択可 |
| プロキシ提供 | 明示禁止はないが、顧客APIキーの利用形態として、Aegis経由を顧客自身が認める形 |
| Zero Data Retention | 一部プランで提供（要確認）。Pro顧客向け差別化に利用可 |

#### Aegisが行うべき対応
- [ ] Anthropic の **DPA / Trust** ページを四半期確認
- [ ] 顧客向け契約に「あなたがAnthropicと結んだDPA上、Aegisを処理委託先として認める」明示
- [ ] Aegisのインフラ・セキュリティ情報を顧客が Anthropic に開示できるよう資料整備
- [ ] エンタープライズ顧客向けに **AWS Bedrock 経由Claude** を推奨ルートとして提示

### 2.2 OpenAI（GPT API）

#### 規約根拠
- **Business Terms**
- **Service Terms**
- **Usage Policies**
- **Enterprise Privacy / DPA**

#### 要点
| 観点 | 内容 |
|---|---|
| API データ学習 | デフォルトで学習に使われない（2023年3月以降） |
| 再販禁止 | "**Resell**" は明示禁止。「顧客のAPIキーを顧客のために中継」は再販に該当しない整理 |
| 第三者処理 | 顧客側の責任で第三者（Aegis）にAPIキーを共有する形 |
| データリージョン | グローバル。Data Residency 契約は Enterprise のみ |
| Zero Data Retention | Enterprise / API契約で zdr 取得可能 |

#### Aegisが行うべき対応
- [ ] 利用規約で「Aegisは顧客のAPIキーを顧客の指示に基づき使用するのみ。再販ではない」明記
- [ ] 顧客に対して「OpenAI 利用規約・使用ポリシーの遵守は顧客の責任」明示
- [ ] OpenAI Trust ページの四半期確認

### 2.3 AWS Bedrock（Claude/Llama/Mistral等）

#### 規約根拠
- **AWS Customer Agreement**
- **AWS Service Terms**
- **Bedrock 特有のモデル提供者規約**（Anthropic, Meta等）

#### 要点
| 観点 | 内容 |
|---|---|
| データ取扱い | AWS内に閉じる。学習利用なし |
| リージョン | 東京リージョン利用可 |
| モデル提供者規約 | Claude on Bedrock は Anthropic の AUP も適用 |
| 再販 | AWS Reseller プログラム経由でない再販は不可。Aegisは「顧客のAWSアカウントを使う」のではなく **顧客が自身のAWSアカウントを持つ前提** |

#### Aegisが行うべき対応
- [ ] **推奨ルート**：顧客が自身のAWSアカウントで Bedrock を有効化し、IAM Role を Aegis に AssumeRole 許可
- [ ] Aegis 自身のAWSアカウントの Bedrock を顧客に共有しない（再販扱いリスク）
- [ ] AWS APN（パートナーネットワーク）登録の検討（信頼性向上）

### 2.4 Google Gemini API（Vertex AI / AI Studio）

#### 規約根拠
- **Google Cloud Terms**
- **Generative AI Service Specific Terms**
- **Gemini API Additional Terms**

#### 要点
| 観点 | 内容 |
|---|---|
| データ取扱い | Vertex AI 経由ならデフォルト学習なし。AI Studio Free 層は学習対象（注意） |
| リージョン | Vertex AI は東京リージョン利用可 |
| 再販 | Google Cloud Partner経由が必要 |

#### Aegisが行うべき対応
- [ ] **AI Studio Free 層の利用は禁止**（顧客のデータが学習に使われ得る）
- [ ] Vertex AI 経由のみサポート、顧客の GCP アカウント利用前提
- [ ] Google Cloud Marketplace 経由の販売は v2 で検討

### 2.5 Azure OpenAI

#### 要点
| 観点 | 内容 |
|---|---|
| データ取扱い | Microsoft内に閉じる。学習利用なし |
| リージョン | Japan East 利用可 |
| 規制対応 | ISMAP、ISO27001、SOC2取得済み（強み） |
| 再販 | CSP/MPN経由が必要 |

#### Aegisが行うべき対応
- [ ] 推奨ルート：顧客が自身の Azure サブスクリプションで Azure OpenAI を有効化、Service Principal を Aegis に共有
- [ ] エンタープライズ・金融・自治体向けには Azure OpenAI を第一推奨

## 3. プロキシ構造の規約適合性チェック

### 3.1 リスクのある構成（避けるべき）

| 構成 | リスク | 理由 |
|---|---|---|
| Aegis 自社の API キーで全顧客の LLM コールを処理 | 高（再販扱い） | 各社の Resale 禁止条項に抵触し得る |
| 学習利用ありの無料層（OpenAI past versions, Gemini AI Studio Free）を本番経路に組込 | 高（顧客データ漏洩リスク） | 顧客プロンプトが学習に使われる |
| LLM プロバイダの規約変更を顧客に通知しない | 中（顧客責任の不履行） | 規約変更は顧客のAPIキー保有者責任 |

### 3.2 推奨構成

```
顧客企業
  └─ 自社で各プロバイダと契約（DPA含む）
      └─ API キーを Aegis に安全に提供（Secrets Manager）
          └─ Aegis は「顧客の指示に基づき」プロキシ
              └─ 各プロバイダAPIへ転送
```

この構造の利点：
- 規約上の当事者関係は「顧客 ⇔ プロバイダ」のまま
- データ取扱責任は顧客が継続的に保有
- Aegisは「処理委託先」として個情法22条の範囲で対応
- プロバイダの規約変更時、影響は顧客の責任

## 4. ユースケース別禁止事項（共通AUP）

各プロバイダの AUP（Acceptable Use Policy）には共通して以下が禁止されている：

| カテゴリ | 例 |
|---|---|
| 違法行為 | 違法薬物、武器、テロ、児童搾取 |
| 個人のプライバシー侵害 | 同意なき身元特定、ストーキング支援 |
| 詐欺・なりすまし | フィッシング、ディープフェイクによる誹謗 |
| 高リスク医療・法律助言 | 医師・弁護士の判断を代替する助言 |
| 選挙妨害 | 投票誘導の自動生成 |
| 差別・暴力扇動 | ヘイト・暴力推奨コンテンツ |

#### Aegis の対応
- 利用規約で「顧客はプロバイダAUPを遵守する義務」を明記
- 重大違反を検知したら一時停止（Guardrails で検出可能なものに限る）
- 違反通報窓口を顧客向け管理画面に設置

## 5. 規約変更追跡プロセス

| 頻度 | 担当 | 内容 |
|---|---|---|
| 月次 | 運用担当 | 各社 Trust / Policy ページのチェック（自動化：Browse the Web → Slack通知） |
| 四半期 | 経営 + 弁護士 | 規約変更が Aegis サービス設計に影響しないかレビュー |
| 緊急 | 全員 | 重大変更時は顧客通知 + 設計変更検討 |

監視対象URL（例、要更新確認）：
- Anthropic: `anthropic.com/legal`, `anthropic.com/trust`
- OpenAI: `openai.com/policies`, `openai.com/enterprise-privacy`
- AWS: `aws.amazon.com/service-terms`, `aws.amazon.com/bedrock`
- Google: `cloud.google.com/terms`, `policies.google.com`
- Microsoft: `learn.microsoft.com/azure/cognitive-services/openai/`

## 6. 顧客への開示・契約上の明記事項

Aegis の利用規約に以下を必ず含める：

1. 顧客は自社で各 LLM プロバイダと契約していること、その規約を遵守する義務
2. Aegis は顧客の APIキーを顧客の指示の範囲で使用する処理委託先である
3. プロバイダの規約変更により、Aegis の機能が変更・停止する可能性
4. AUP違反コンテンツを Aegis が検知した場合の対応（一時停止等）
5. プロバイダから顧客への請求は Aegis の関知するところではない
6. 顧客がプロバイダから契約解除された場合、Aegis サービスも同時停止し得る

## 7. リスクサマリー

| リスク | 確率 | 影響 | 対応 |
|---|---|---|---|
| プロバイダが「プロキシ禁止」を明文化 | 低 | 致命的 | 早期検知。各社事業窓口に事前相談（特にエンタープライズ営業窓口） |
| プロバイダが顧客契約を解除→Aegis経由の利用も停止 | 中 | 高 | フェイルオーバーで他プロバイダへ自動切替 |
| プロバイダがAPI価格を急変更 | 高 | 中 | 顧客への即時通知、複数プロバイダ前提 |
| AUP違反を Aegis が見逃して提供加担 | 中 | 高 | Guardrails 標準導入、検知ログ保全 |
| 顧客APIキー漏洩 | 低 | 致命的 | KMS Envelope、ローテーション、SOC2準拠 |
